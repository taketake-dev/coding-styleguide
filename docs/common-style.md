# コーディング共通のスタイル

## コーディングスタイル

- UTF8
- 改行はLF
- ファイルの最後に空行を1行置く

## コメント

- 基本的に自由
- AI特有のフレンドリーな表現や絵文字は不要

## Print、画面への表示

- 使用できる状態であれば極力loggingを使うこと
- 日本語でよい
- 絵文字は見やすさや分類・区分けのためであれば少々使用してよい
- フレンドリーな表現は避ける

### 文章のルール

- 言語は「日本語（です・ます調）」にする。
- 形式は [機能/対象] 動作の結果 (補足情報) に固定する。
- 感情的な表現は使わず、事実だけを書く。

```code
logger.info("[Auth] ユーザーのログインに成功しました (user_id=9876)")
logger.error("[File] 設定ファイルの読み込みに失敗しました (path=config.json)")
```

### loggingのルール

出力する情報の重要度に応じて以下の5つのログレベルを使い分けること。

- DEBUG：開発・デバッグ時のみ必要な詳細情報（変数の推移や内部状態など）
- INFO：システムの正常な運用・動向の記録（「サービスの起動」「処理の完了」など）
- WARNING：現在は問題ないが、将来的にエラーに繋がる可能性がある予兆（「ディスク容量の逼迫」「非推奨機能の使用」など）
- ERROR：システムの一部で処理が失敗した状態（「特定のAPI呼び出しの失敗」「ファイルの読み込み失敗」など。システム自体は稼働を継続できるもの）
- CRITICAL / FATAL：システム全体が停止するような致命的な障害（「データベースへの接続不能」「メモリ枯渇による強制終了」など）

以下の4つの要素を同一の並び順で含めるように。

- タイムスタンプ（日時）
  - 形式：ISO 8601形式（例：YYYY-MM-DDTHH:mm:ss.sss）を推奨。
  - 精度：ミリ秒単位まで必須。
- ログレベル（重要度）
  - 形式：大文字表記（DEBUG, INFO, ERROR 等）で統一。
- ソースコンテキスト（出力元）
  - 形式：ログを出力した「モジュール名」「クラス名」「ファイル名と行数」のいずれかを明記。
- メッセージ（内容）
  - 形式：発生した事実（および必要に応じてID等の識別子）を簡潔に記述。

## Git commit

Conventional Commits を使用する。プレフィックスは以下の通り：

| プレフィックス | 説明 |
| --- | --- |
| feat | 新しい機能 |
| fix | バグ修正 |
| docs | ドキュメント関連（README、コメント等） |
| style | コード形式の変更（ロジック変更なし） |
| refactor | リファクタリング |
| perf | パフォーマンス改善 |
| test | テスト追加・修正 |
| chore | ビルド設定、依存関係、ツール類 |
| ci | CI/CD設定変更 |

## .envの書き方

.env.exampleを別途作る

```.env.example
# ==============================================================================
# 使い方: このファイルをプロジェクトのルートディレクトリに `.env` という名前でコピーし、
#        環境に応じた正しい値を設定してください。
# コマンド: cp .env.example .env
# ==============================================================================

# モデルファイルの絶対パス
MODEL_PATH=path/to/your/model.safetensors

# 画像の出力ディレクトリ
OUTPUT_DIR=output
```
